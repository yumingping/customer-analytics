# 数据库设计文档（修订版）

> 修订日期：2026-07-15 | 沙箱版本：v2（全量 54 列）

## 概述

本系统采用三表分层架构，基于业务主键 `member_no` 进行关联，避免使用自增 ID 和物理外键，以提升 ETL 效率和数据合并灵活性。数据源为航空公司客户 `air.csv`（62988 行 × 44 原始字段），经清洗、特征工程、RFM 计算、K-Means 聚类后拆分入库。

## 架构设计

### 设计理念

1. **业务主键驱动**：所有表使用 `member_no`（会员编号）作为主键，便于数据追踪和跨表关联
2. **三表分层**：维度表（Base）→ 事实表（Summary）→ 应用表（Analytics），职责清晰
3. **无物理外键**：表间关系通过应用层 JOIN 实现，避免外键约束导致的写入失败
4. **双引擎支持**：MySQL（生产）优先，不可用时自动降级到 SQLite 沙箱

## 表结构详解

### 1. customer_base（客户基础信息表）

**用途**：存储客户静态档案信息，变更频率低。

| 字段名 | 类型 | 可空 | 说明 |
|--------|------|------|------|
| member_no | VARCHAR(20) | NO | 会员编号（业务主键） |
| age | TINYINT | YES | 年龄（已处理异常值，范围 1-100） |
| gender | VARCHAR(10) | YES | 性别（男/女/未知） |
| ffp_tier | TINYINT | YES | 会员等级（4/5/6 等） |
| avg_discount | DECIMAL(5,2) | YES | 平均折扣率 |
| ffp_date | DATE | YES | 入会日期 |
| first_flight | DATE | YES | 首次飞行日期 |
| work_province | VARCHAR(20) | YES | 工作所在省份（如 广东省、北京市） |
| work_city | VARCHAR(20) | YES | 工作所在城市 |
| work_country | VARCHAR(10) | YES | 工作所在国家 |
| load_time | DATE | YES | 数据观察截止时间（2014-03-31） |
| membership_years | FLOAT | YES | 入会时长（年），= (观察截止日 - FFP_DATE) / 365.25 |

**索引**：
- PRIMARY KEY (member_no)
- INDEX idx_gender_tier (gender, ffp_tier)

**索引**：
- PRIMARY KEY (member_no)

### 3. customer_analytics（分析结果表）

**用途**：存储 RFM 评分、聚类标签和价值分层结果，由算法流水线写入。

| 字段名 | 类型 | 可空 | 说明 |
|--------|------|------|------|
| member_no | VARCHAR(20) | NO | 会员编号（关联键） |
| r_score | TINYINT | YES | 最近消费评分（1-5，百分位法） |
| f_score | TINYINT | YES | 消费频率评分（1-5，百分位法） |
| m_score | TINYINT | YES | 消费金额评分（1-5，百分位法） |
| rfm_total | TINYINT | YES | RFM 总分（3-15） |
| cluster | TINYINT | YES | 聚类标签（0-3，默认 K=4） |
| value_label | VARCHAR(20) | YES | 价值标签（高价值/中价值/低价值/流失客户） |

**价值分层规则**：

| 标签 | rfm_total 范围 |
|------|---------------|
| 高价值客户 | ≥ 13 |
| 中价值客户 | 9 ~ 12 |
| 低价值客户 | 6 ~ 8 |
| 流失客户 | < 6 |

**索引**：
- PRIMARY KEY (member_no)
- INDEX idx_cluster (cluster)
- INDEX idx_rfm_label (rfm_total, value_label)

### 4. customer_rfm（宽表视图）

**用途**：三表 1:1 JOIN 的宽表（54 列），包含上述所有字段。查询时可直接 `SELECT * FROM customer_rfm` 免写 JOIN。

## SQL 查询规范

### 表别名规则

在编写 SQL 查询时，必须使用以下标准别名：

| 表名 | 别名 |
|------|------|
| customer_base | b |
| customer_flight_summary | f |
| customer_analytics | a |

### 查询示例

**示例 1：按省份统计客户数量**

```sql
SELECT b.work_province, COUNT(*) AS cnt
FROM customer_base b
GROUP BY b.work_province
ORDER BY cnt DESC;
```

**示例 2：查询高价值客户的平均年龄**

```sql
SELECT 
    AVG(b.age) as avg_age,
    COUNT(*) as count
FROM customer_base b
JOIN customer_analytics a ON b.member_no = a.member_no
WHERE a.value_label = '高价值客户';
```

**示例 3：各聚类飞行里程对比**

```sql
SELECT 
    a.cluster,
    AVG(f.seg_km_sum) AS avg_km,
    AVG(f.flight_count) AS avg_flights
FROM customer_analytics a
JOIN customer_flight_summary f ON a.member_no = f.member_no
GROUP BY a.cluster;
```

**示例 4：年龄区间分布（使用 CASE WHEN）**

```sql
SELECT 
    CASE 
        WHEN b.age < 25 THEN '25岁以下'
        WHEN b.age BETWEEN 25 AND 35 THEN '25-35岁'
        WHEN b.age BETWEEN 36 AND 45 THEN '36-45岁'
        WHEN b.age BETWEEN 46 AND 55 THEN '46-55岁'
        ELSE '55岁以上'
    END as age_group,
    COUNT(*) as count
FROM customer_base b
GROUP BY age_group
ORDER BY count DESC;
```

**示例 5：近一年高活跃客户占比**

```sql
SELECT 
    CASE WHEN f.l1y_flight_count >= 10 THEN '高活跃' ELSE '一般' END AS level,
    COUNT(*) AS cnt
FROM customer_flight_summary f
GROUP BY level;
```

**示例 6：各会员等级的平均飞行间隔**

```sql
SELECT 
    b.ffp_tier,
    AVG(f.avg_interval) AS avg_interval_days,
    AVG(f.max_interval) AS max_interval_days
FROM customer_base b
JOIN customer_flight_summary f ON b.member_no = f.member_no
GROUP BY b.ffp_tier
ORDER BY b.ffp_tier;
```

## 数据流转

### 1. 数据清洗阶段

```
原始数据 (air.csv)
    ↓
data_cleaning.py
    ↓
清洗后数据 (62987 条)
```

**清洗规则**：
- AGE 缺失值填充为中位数
- avg_discount 缺失值填充为均值
- GENDER 缺失值填充为"未知"
- 剔除飞行次数/里程 ≤ 0 的记录
- 剔除年龄 ≤ 0 或 > 100 的记录

### 2. RFM 计算阶段

```
清洗后数据
    ↓
rfm_analysis.py
    ↓
RFM 特征 + 评分
```

**计算逻辑**：
- **R (Recency)**：距观察期末天数，越小越活跃
- **F (Frequency)**：总飞行次数
- **M (Monetary)**：总飞行里程

**评分规则**（百分位法）：
- 0-20% → 1 分
- 20-40% → 2 分
- 40-60% → 3 分
- 60-80% → 4 分
- 80-100% → 5 分

### 3. 聚类分析阶段

```
RFM 特征
    ↓
Log1p + StandardScaler（预处理）
    ↓
K-Means 聚类（K 值由 Silhouette+SSE 双指标自动寻优）
    ↓
聚类标签 + 价值分层
```

### 4. 数据入库阶段

```
最终数据 (54 列)
    ↓
database.py / sql_tools.py
    ↓
MySQL（优先）/ SQLite 沙箱（降级）
```

**写入顺序**：
1. customer_base（基础信息，12 列）
2. customer_flight_summary（飞行统计，37 列）
3. customer_analytics（分析结果，7 列）
4. customer_rfm（宽表，54 列）

## 性能优化

### 1. 索引策略

- **customer_base**：(gender, ffp_tier) 复合索引，加速人群画像查询
- **customer_flight_summary**：recency 索引，加速活跃度查询
- **customer_analytics**：cluster 索引 + (rfm_total, value_label) 复合索引，加速聚类查询

### 2. 分页机制

查询结果超过 20 行时，系统自动启用分页：

```python
# app.py 中的分页逻辑
page = request.json.get('page', 1)
page_size = request.json.get('page_size', 20)
offset = (page - 1) * page_size
```

### 3. 缓存策略

- 查询结果缓存 10 分钟（_QUERY_CACHE）
- 支持翻页时复用缓存
- 缓存最大 50 条，TTL 600 秒

## 常见问题

### Q1: 为什么不使用自增 ID？

**A**: 业务主键 `member_no` 已具备唯一性，使用自增 ID 会导致：
- 数据合并时主键冲突
- ETL 写入效率降低
- 跨系统数据同步困难

### Q2: 为什么禁用物理外键？

**A**: 物理外键约束会：
- 降低批量写入速度
- 导致外键死锁
- 增加数据迁移复杂度

应用层通过 JOIN 实现逻辑关联，更灵活高效。

### Q3: 如何修改聚类数量？

**A**: 编辑 `config.py`：

```python
N_CLUSTERS = 4          # 修改默认聚类数
CLUSTER_RANGE = range(2, 9)  # 修改 K 值搜索范围
```

重新运行 `python main.py` 即可。

### Q4: 为什么查询结果被分页？

**A**: 防止前端渲染卡顿。超过 20 行的结果自动分页，可通过翻页按钮浏览全部数据。

### Q5: MySQL 和 SQLite 有什么区别？

**A**: 系统优先连接 MySQL，连接失败自动降级到 SQLite 沙箱。两者表结构和数据完全一致，SQL 查询语法略有差异（日期函数、反引号等），系统会自动适配。
