# 数据库设计文档

## 概述

本系统采用三表分层架构，基于业务主键 `member_no` 进行关联，避免使用自增 ID 和物理外键，以提升 ETL 效率和数据合并灵活性。

## 架构设计

### 设计理念

1. **业务主键驱动**：所有表使用 `member_no`（会员编号）作为主键，便于数据追踪和跨表关联
2. **三表分层**：维度表（Base）→ 事实表（Summary）→ 应用表（Analytics），职责清晰
3. **无物理外键**：表间关系通过应用层 JOIN 实现，避免外键约束导致的写入失败
4. **时间戳审计**：所有表包含 `create_time` 和 `update_time`，支持数据溯源

## 表结构详解

### 1. customer_base（客户基础信息表）

**用途**：存储客户静态档案信息，变更频率低。

| 字段名 | 类型 | 可空 | 说明 |
|--------|------|------|------|
| member_no | VARCHAR(20) | NO | 会员编号（业务主键） |
| age | TINYINT | YES | 年龄（已处理异常值） |
| gender | VARCHAR(10) | YES | 性别（男/女/未知） |
| ffp_tier | TINYINT | YES | 会员等级（4/5/6 等） |
| avg_discount | DECIMAL(5,2) | YES | 平均折扣率 |
| ffp_date | DATE | YES | 入会日期 |
| first_flight | DATE | YES | 首次飞行日期 |
| create_time | DATETIME | NO | 记录创建时间 |
| update_time | DATETIME | NO | 记录更新时间 |

**索引**：
- PRIMARY KEY (member_no)
- INDEX idx_gender_tier (gender, ffp_tier)

### 2. customer_flight_summary（飞行行为事实表）

**用途**：存储观察期内客户的飞行行为统计数据。

| 字段名 | 类型 | 可空 | 说明 |
|--------|------|------|------|
| member_no | VARCHAR(20) | NO | 会员编号（关联键） |
| flight_count | INT | YES | 总飞行次数（F 特征） |
| seg_km_sum | INT | YES | 总飞行里程（M 特征） |
| bp_sum | INT | YES | 总基本积分 |
| last_flight | DATE | YES | 最后一次飞行日期 |
| recency | INT | YES | 距观察期末天数（R 特征，越小越活跃） |
| create_time | DATETIME | NO | 记录创建时间 |
| update_time | DATETIME | NO | 记录更新时间 |

**索引**：
- PRIMARY KEY (member_no)
- INDEX idx_recency (recency)

### 3. customer_analytics（分析结果表）

**用途**：存储 RFM 评分、聚类标签和价值分层结果，由算法流水线写入。

| 字段名 | 类型 | 可空 | 说明 |
|--------|------|------|------|
| member_no | VARCHAR(20) | NO | 会员编号（关联键） |
| r_score | TINYINT | YES | 最近消费评分（1-5，百分位法） |
| f_score | TINYINT | YES | 消费频率评分（1-5，百分位法） |
| m_score | TINYINT | YES | 消费金额评分（1-5，百分位法） |
| rfm_total | TINYINT | YES | RFM 总分（3-15） |
| cluster | TINYINT | YES | 聚类标签（0-3，默认 K=4） |
| value_label | VARCHAR(20) | YES | 价值标签（高价值/中价值/低价值/流失客户） |
| update_time | DATETIME | NO | 算法计算更新时间 |

**索引**：
- PRIMARY KEY (member_no)
- INDEX idx_cluster (cluster)
- INDEX idx_rfm_label (rfm_total, value_label)

## SQL 查询规范

### 表别名规则

在编写 SQL 查询时，必须使用以下标准别名：

| 表名 | 别名 |
|------|------|
| customer_base | b |
| customer_flight_summary | f |
| customer_analytics | a |

### 禁用字段列表

以下字段在数据库中**不存在**，禁止在 SQL 中使用：

- `city`（城市）
- `工作地点`（工作地点）
- `姓名`（客户姓名）
- `电话`（联系电话）

### 查询示例

**示例 1：查询高价值客户的平均年龄**

```sql
SELECT 
    AVG(b.age) as avg_age,
    COUNT(*) as count
FROM customer_base b
JOIN customer_analytics a ON b.member_no = a.member_no
WHERE a.value_label = '高价值客户';
```

**示例 2：查询各聚类的客户数量分布**

```sql
SELECT 
    a.cluster,
    a.value_label,
    COUNT(*) as count
FROM customer_analytics a
GROUP BY a.cluster, a.value_label
ORDER BY a.cluster;
```

**示例 3：查询飞行里程 Top 10 的客户**

```sql
SELECT 
    b.member_no,
    b.age,
    b.gender,
    f.seg_km_sum,
    a.value_label
FROM customer_base b
JOIN customer_flight_summary f ON b.member_no = f.member_no
JOIN customer_analytics a ON b.member_no = a.member_no
ORDER BY f.seg_km_sum DESC
LIMIT 10;
```

**示例 4：不同会员等级的 RFM 得分对比**

```sql
SELECT 
    b.ffp_tier,
    AVG(a.r_score) as avg_r,
    AVG(a.f_score) as avg_f,
    AVG(a.m_score) as avg_m,
    AVG(a.rfm_total) as avg_total
FROM customer_base b
JOIN customer_analytics a ON b.member_no = a.member_no
GROUP BY b.ffp_tier
ORDER BY b.ffp_tier;
```

**示例 5：年龄区间分布（使用 CASE WHEN）**

```sql
SELECT 
    CASE 
        WHEN b.age < 25 THEN '25岁以下'
        WHEN b.age BETWEEN 25 AND 35 THEN '25-35岁'
        WHEN b.age BETWEEN 36 AND 45 THEN '36-45岁'
        WHEN b.age BETWEEN 46 AND 55 THEN '46-55岁'
        ELSE '55岁以上'
    END as age_group,
    COUNT(*) as count
FROM customer_base b
GROUP BY age_group
ORDER BY count DESC;
```

## 数据流转

### 1. 数据清洗阶段

```
原始数据 (air.csv)
    ↓
data_cleaning.py
    ↓
清洗后数据 (62987 条)
```

**清洗规则**：
- 去除重复记录
- 处理异常值（如年龄 > 100）
- 填充缺失值（gender 默认为"未知"）

### 2. RFM 计算阶段

```
清洗后数据
    ↓
rfm_analysis.py
    ↓
RFM 特征 + 评分
```

**计算逻辑**：
- **R (Recency)**：距观察期末天数，越小越活跃
- **F (Frequency)**：总飞行次数
- **M (Monetary)**：总飞行里程

**评分规则**（百分位法）：
- 0-20% → 1 分
- 20-40% → 2 分
- 40-60% → 3 分
- 60-80% → 4 分
- 80-100% → 5 分

### 3. 聚类分析阶段

```
RFM 特征
    ↓
K-Means (K=4)
    ↓
聚类标签 + 价值分层
```

**聚类结果映射**：
- Cluster 0 → 高价值客户（RFM 总分高）
- Cluster 1 → 中价值客户（RFM 总分中等）
- Cluster 2 → 低价值客户（RFM 总分低）
- Cluster 3 → 流失客户（Recency 高，活跃度低）

### 4. 数据入库阶段

```
三表数据
    ↓
database.py
    ↓
MySQL / SQLite
```

**写入顺序**：
1. customer_base（基础信息）
2. customer_flight_summary（飞行统计）
3. customer_analytics（分析结果）

## 性能优化

### 1. 索引策略

- **customer_base**：(gender, ffp_tier) 复合索引，加速人群画像查询
- **customer_flight_summary**：recency 索引，加速活跃度查询
- **customer_analytics**：cluster 索引 + (rfm_total, value_label) 复合索引，加速聚类查询

### 2. 分页机制

查询结果超过 20 行时，系统自动启用分页：

```python
# app.py 中的分页逻辑
page = request.json.get('page', 1)
page_size = request.json.get('page_size', 20)
offset = (page - 1) * page_size

# SQL 查询
sql = f"{base_sql} LIMIT {page_size} OFFSET {offset}"
```

### 3. 缓存策略

- 查询结果缓存 10 分钟（_QUERY_CACHE）
- 支持翻页时复用缓存
- 缓存键：query_id（基于 SQL 哈希）

## 常见问题

### Q1: 为什么不使用自增 ID？

**A**: 业务主键 `member_no` 已具备唯一性，使用自增 ID 会导致：
- 数据合并时主键冲突
- ETL 写入效率降低
- 跨系统数据同步困难

### Q2: 为什么禁用物理外键？

**A**: 物理外键约束会：
- 降低批量写入速度
- 导致外键死锁
- 增加数据迁移复杂度

应用层通过 JOIN 实现逻辑关联，更灵活高效。

### Q3: 如何修改聚类数量？

**A**: 编辑 `config.py`：

```python
N_CLUSTERS = 4  # 修改为所需数量
```

重新运行 `python main.py` 即可。

### Q4: 为什么查询结果被分页？

**A**: 防止前端渲染卡顿。超过 20 行的结果自动分页，可通过翻页按钮浏览全部数据。

### Q5: 树图显示拥挤怎么办？

**A**: 系统已启用自动扁平化（Top 15 + Other）。若仍有问题，调整 `chart_tools.py` 中的 `leafDepth` 参数。
